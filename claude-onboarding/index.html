대상: ChatGPT/Claude를 가볍게 써본 비개발자 실무자

오늘의 목표

강의가 끝나면 다음 5가지를 본인 업무 언어로 말할 수 있다.

  • Claude.ai · Desktop · Code · Cowork가 각각 어디에 쓰이는지 구분한다.
  • 파일 업로드만으로 분석/문서 산출물을 직접 설계한다.
  • Project · MCP · Skill이 각각 무엇을 늘려주는지 한 줄로 설명한다.
  • 반복 업무를 "프롬프트 한 번"이 아니라 "데이터 + 작업 규칙 + 산출물 형식"으로 본다.
  • 사내 보고서 Skill 초안을 만들기 위한 재료를 모은다.

Claude 제품 지형도

하나의 도구가 아니라 3층으로 나눠서 본다.

1층 · 인터페이스

Claude.ai, Desktop, Code, Cowork

2층 · 기본 능력

무엇을 하나?

대화 · 요약 · 파일 분석 · 문서 생성 · 자동화

3층 · 확장

어떻게 강해지나?

Projects · MCP · Skills · Plugins

"같은 모델이지만 어디서 어떻게 쓰느냐가 다 다르다."

어디서 쓰나 — 4가지 진입점

제품한 줄 비유가장 먼저 쓰는 상황
Claude.ai가장 먼저 써볼 곳파일 업로드 + 문서/분석
Claude Desktop로컬 앱 중심 사용내 Mac/PC 안의 파일·툴 연결
Claude CodeVBA 바깥의 자동화 엔진폴더·파일·브라우저를 넘나드는 작업
Claude Cowork조직용 협업 에이전트 운영팀 전체가 같은 Skill을 공유

오늘 데모는 모두 Claude.ai 한 곳에서 진행한다. 나머지는 "이런 게 가능하다" 수준으로만 본다.

업무 프롬프트의 5요소

"좋은 프롬프트"보다 좋은 입력 자료 + 명시된 기준이 결과를 만든다.

1. 목표

무엇을 결정하려는가?

2. 자료

어떤 파일/숫자를 줄 것인가?

3. 기준

어떤 방식으로 판단할 것인가?

4. 형식

어떤 산출물로 받을 것인가?

5. 검증

무엇을 다시 확인할 것인가?

숫자 출처 · 가정 · 리스크는 별도로 표기하게 한다.

데모 1 — 신제품 출시 데이터 분석

가상회사 모닝브루랩 · 무가당 RTD 콜드브루 브루핏 라이트.

업로드 파일
  • company_context.md
  • sales_by_channel.csv
  • customer_survey.csv
  • competitor_landscape.csv
받을 산출물
  • 대표용 1페이지 분석 메모
  • 5장 발표자료 목차
  • 투자자용 스토리 변형

같은 데이터, 독자만 바뀌어도 결과가 어떻게 달라지는지를 보여준다.

데모 1 — 분석 결과의 모양

실제로 받을 결과물의 골격은 다음과 같다.

1페이지 메모
  • 이번 시점 핵심 판단 3개
  • 채널별 성과 비교 (자사몰 / 편의점 / B2B)
  • 고객 구매 이유 / 불만 패턴
  • 경쟁사 대비 포지션
  • 추가로 확인해야 할 데이터
5장 발표자료 목차
  • 슬라이드 제목
  • 핵심 메시지 1문장
  • 들어갈 표/그래프
  • 발표자 코멘트

중요한 건 결과 자체가 아니라 "결정 가능한 형태로 받게 만드는 요청 방식"이다.

MCP — 외부 도구 연결

Claude가 외부 도구·데이터에 접근하는 연결 규격이다.

예시
  • Google Drive에서 파일 찾기
  • Slack 메시지 검색
  • Gmail 초안 작성
  • 사내 DB 조회
주의
  • 권한을 "주는 것"이다 → 보안 검토 필수
  • 플랜·조직 정책에 따라 가능 범위가 다름
  • 라이브 시연은 리스크 → 구조 설명으로 대체

Skill — 일하는 방식의 재사용

우리 회사 일하는 법을 폴더로 저장한 것이다.

skill_demo/reporting_skill/
├── SKILL.md             ← 작업 절차, 품질 기준, 금지 항목
├── report_template.md   ← 우리 회사 보고서 양식
└── style_guide.md       ← 말투, 선호/회피 표현, 숫자 표기 규칙

Skill = "보고서 양식 + 작업 절차 + 말투 가이드"를 한 묶음으로 둔 것.
다음에도 같은 결과를 반복해서 받기 위해 만든다.

MCP + Skill 조합

MCP만

Claude가 Google Drive 파일을 찾고 읽을 수 있다.

Skill만

Claude가 우리 회사 주간보고 형식을 알고 있다.

MCP + Skill

Drive에서 이번 주 자료를 찾아 우리 형식으로 정리한다.

MCP가 손이라면, Skill은 일하는 방법이다. 둘이 합쳐져야 "내 업무"가 된다.

데모 2 — 주간보고 Skill 맛보기

업로드 파일
  • weekly_kpi.csv (14~17주차)
  • support_tickets.csv
  • field_notes.md
  • SKILL.md + 템플릿 + 스타일 가이드
받을 산출물
  • 대표 보고용 주간보고 1장
  • 대표 결정 안건 3개
  • 회의 아젠다 변환본

핵심: 결과의 일관성. 다음 주에도, 누가 작성하든, 같은 톤으로 나온다.

커뮤니티 사례 — 짧게만

gstack

기획·리뷰·QA·릴리스를 역할별 워크플로로 묶은 Skill 모음. 소프트웨어 전달 과정용.

ultraplan

요구사항 질문 → PRD → 실행 계획을 클라우드에서 작성. /ultraplan으로 호출.

개발 중심 사례지만, 핵심 메시지는 같다. "업무 절차를 패키지화한다."

내 업무에 적용하는 4단계

1

이번 주 반복 문서 하나를 고른다.

주간보고, 회의록, 영업 일지, 리포트 요약 등.

2

좋은 샘플 2개 + 나쁜 샘플 1개를 모은다.

"우리가 원하는 결과"의 구체적 모양을 Claude에 보여주기 위한 재료.

3

지켜야 할 양식·말투·판단 기준을 적는다.

SKILL.md 한 페이지로. 길게 안 써도 된다.

4

Claude에게 "이 문서용 Skill 초안을 만들어줘"라고 요청한다.

Claude가 빠진 항목을 되묻게 만든다. 그게 곧 내가 미처 정리 못 한 규칙이다.
